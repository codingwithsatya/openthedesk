Week 1 complete — 8 / 8 milestones
Core LLM concepts
Tokens
~¾ of a word. Unit of cost and context. Claude = 200K token window. You pay per input + output token.
Embeddings
Meaning encoded as numbers (vectors). Similar meaning = similar vectors. Foundation of RAG in Month 2.
Temperature
0 = deterministic. 0.5 = balanced. 1.0 = creative. Use low (0–0.3) for structured/JSON output.
Stateless API
No memory between calls. You send full conversation history every single time. Managing this = key engineering challenge.
System vs user prompt
System = persistent rules (role, format, constraints). User = actual input. Put constraints in system prompt.
Prompt engineering patterns

Prompt structure

Role + Context + Task + Format. Always define all four layers for reliable, consistent output.

Chain-of-thought

Add "think step by step" before complex tasks. Dramatically improves accuracy on reasoning problems.

Few-shot learning

Show 2–3 input/output examples. Model matches the pattern without needing long instructions.

Structured output

"Respond with valid JSON only. No markdown. No backticks." Then use json.loads() to parse.

Edge case handling

Always define what to return when data is missing. "If no phone number, return null." Else model hallucinates.

Prompts are code

Test on 50 inputs. Version control them. Handle errors. Treat prompts exactly like software functions.

Conversation memory
Maintain a messages list yourself. Append user + assistant turns every call. The model has no memory — you do.
Streaming responses
Use client.messages.stream() instead of create(). Yields text chunks in real time. Makes apps feel alive vs waiting for full response.
Token cost growth per turn

Context window guard

Week 3 complete — 8 / 8 milestones
Core concepts
Persona + Constraint + Format trifecta
The most reliable system prompt structure. Persona sets expertise, constraints define what the model won't do, format locks the output shape. All three together = consistent production output.
Structured output as a contract
Say it 3 ways: "ONLY valid JSON. No prose. No markdown. No backticks." Give the exact schema. Define the null case explicitly ("if no issues, return []"). Missing any one of these causes KeyErrors in prod.
XML tags for complex inputs
Wrap user-submitted code in <code> tags. Prevents the model confusing instructions with content. Especially important when code contains prompt-like strings.
Temperature 0.1 for structured output
Use near-zero temperature when parsing JSON. Creative temperature randomly changes schema shape. 0.1 gives near-deterministic output while avoiding the occasional freeze of 0.
Prompts as functions — engineering mindset
Function signature = input schema. Return type = JSON shape. Error handling = fallback instructions. Unit tests = 20–30 fixed inputs. Version control prompts in git like code.
Global env var override pitfall
load_dotenv() only sets a variable if it's NOT already in the environment. A stale global key in ~/.zshrc silently wins over .env. Fix: unset ANTHROPIC_API_KEY in terminal, or remove from shell config.

Core concepts
Supabase pgvector setup
Enable vector extension. Create documents + chunks tables. Build match_chunks() SQL function with cosine distance operator <=>. user_id filter for data isolation.
Voyage AI embeddings
voyage.embed(texts, model="voyage-3", input_type="document"). Returns 1024-dim vectors. input_type="query" for search queries. Batch up to 128 texts per call. Free tier = 3 RPM.
PDF text extraction
PyPDF2.PdfReader. Extract page by page. Some PDFs have no text layer (scanned) — PyPDF2 returns empty. Text-only RAG works for 80% of real documents.
Chunking with overlap
Split by word count (CHUNK_SIZE=400, CHUNK_OVERLAP=50). Overlap prevents key sentences being split across chunks. Skip chunks under 50 chars — they're noise.
Ingestion pipeline
PDF → extract pages → chunk text → embed chunks (Voyage) → store in Supabase with doc_id, user_id, page_number, chunk_index. Insert in batches of 50.
Retrieval pipeline
Embed query (Voyage, input_type="query") → supabase.rpc("match_chunks") → get top 5 chunks → build context string → Claude answers with citation constraint.
Confidence threshold
match_threshold=0.3 in match_chunks(). If 0 chunks returned, respond "I don't have that information." Never let Claude answer from empty context — it hallucinates.
File upload endpoint
FastAPI UploadFile + python-multipart. Save to tempfile, ingest, delete. Return doc_id, pages, chunks count. /ask endpoint embeds question + retrieves + asks Claude.
Code patterns
Supabase vector search function
-- SQL function in Supabase create or replace function match_chunks( query_embedding vector(1024), match_user_id text, match_count int default 5, match_threshold float default 0.5 ) returns table (id uuid, content text, page_number int, similarity float) as $$ select id, content, page_number, 1 - (embedding <=> query_embedding) as similarity from chunks where user_id = match_user_id and 1 - (embedding <=> query_embedding) > match_threshold order by embedding <=> query_embedding limit match_count; $$ language sql stable;

Week 4 complete — 7 / 7 milestones · Month 1 complete 🎉
Core concepts
Prompt caching
Add cache_control: ephemeral to your system prompt. Claude caches it after the first call. Subsequent calls cost 90% less on those tokens. One line change — biggest cost lever in production.
Batch processing
Submit multiple inputs in one API call. Returns individual results + cross-file summary. The summary object (avg score, total issues, highest risk) is what dashboards and managers actually read.
Model routing by task complexity
Haiku = 10x cheaper for simple classification. Sonnet = best cost/quality for reasoning. Opus = complex multi-step agents. Routing cheap tasks to Haiku cuts your bill 60–70% at scale.
Cost math at scale
10K calls/day × 500 token system prompt = 5M input tokens/day on system prompt alone. With caching: ~500K tokens/day. That's the difference between $8/day and $0.80/day just on the system prompt.
Component decomposition in React
Split SingleAnalyzer, BatchAnalyzer, and ResultCard into separate components. ResultCard is shared between both. Keeps the file readable and prevents duplicating the severity card + suggestions logic.

Week 7 complete — 6 / 6 milestones · Advanced RAG pipeline shipped 🎉
Core concepts
Hybrid search — semantic + keyword combined
Semantic search finds meaning. Keyword search (BM25) finds exact terms. Hybrid combines both with configurable weights (50/50 for OpenTheDesk). Fixes the PTR-FAST vs PTR-FULL confusion problem.
BM25 keyword search
Best Match 25 algorithm — counts word frequency weighted by document length. rank_bm25 library in Python. Tokenize corpus once, score all docs against query. Returns 0 for no match (expected for conceptual queries).
Reciprocal Rank Fusion (RRF)
Merges two ranked lists into one. Formula: score += 1/(60 + rank) for each list. Items ranked high in both lists bubble to top. k=60 constant prevents extreme ranks from dominating. No score normalization needed.
Reranking with Cohere
After hybrid search returns 20 candidates, Cohere rerank-english-v3.0 reads query + each chunk together and scores true relevance. 0.99 = extremely confident. ~$1/1000 queries. Use top_n=5 from 20 candidates.
Confidence threshold
Check rerank_score of top result. Below 0.1 = refuse to answer. Above 0.65 = answer confidently. Prevents confident wrong answers from irrelevant chunks. Critical for OpenTheDesk — wrong rule during live trade = bad trade.
Python-side hybrid search (free tier workaround)
Supabase free tier blocks GIN indexes (memory limit 32MB). Solution: semantic search via pgvector, BM25 keyword search in Python over retrieved chunks, RRF merge in Python. Cleaner architecture anyway — more control.
Code patterns
Full hybrid search pipeline
def hybrid_search_pipeline(question, top_k=5): # 1. Embed question vector = voyage.embed([question], model="voyage-3", input_type="query") # 2. Semantic search — top 20 from pgvector semantic = semantic_search(vector, top_k=20) # 3. BM25 keyword search over same chunks keyword = keyword_search(question, semantic, top_k=20) # 4. RRF merge fused = reciprocal_rank_fusion(semantic, keyword) # 5. Rerank top candidates return rerank(question, fused[:20], top_n=top_k)
BM25 keyword search in Python
from rank_bm25 import BM25Okapi def keyword_search(query, chunks, top_k=20): tokenized = [c["content"].lower().split() for c in chunks] bm25 = BM25Okapi(tokenized) scores = bm25.get_scores(query.lower().split()) ranked = sorted(enumerate(scores), key=lambda x: x[1], reverse=True) return [chunks[i] for i, score in ranked[:top_k] if score > 0]
Cohere reranking
import cohere co = cohere.Client(api_key=os.environ["COHERE_API_KEY"]) def rerank(query, chunks, top_n=5): docs = [c["content"] for c in chunks] response = co.rerank( model="rerank-english-v3.0", query=query, documents=docs, top_n=top_n ) return [chunks[r.index] | {"rerank_score": r.relevance_score} for r in response.results]

yfinance for live market data
Free, no API key needed. ^GSPC = SPX, ^VIX = VIX. history(period="5d") gets last 5 trading days. Last row = most recent completed day. During market hours today's data is incomplete — PDC = second-to-last row. Correct behavior by design.
Saty ATR levels auto-calculated
calculate_atr_levels(pdc, atr) computes all 8 levels: ±23.6% trigger, ±38.2% GG open, ±61.8% GG complete, ±100% full ATR. ATR approximated from prior day range (H-L). Saty ATR indicator gives exact value — use that to verify.
PREMARKET endpoint with live data injection
POST /premarket fetches live SPX + VIX + ATR levels, injects into user message as structured context, streams Claude response. Agent receives real PDC, PDH, PDL, VIX alongside your trading rules. PREMARKET button in UI calls this endpoint directly.
Polygon free tier limitations
Polygon free tier doesn't include SPX index data or options chain snapshots. yfinance is the better free alternative for SPX/VIX. Options data requires Polygon paid tier ($29/mo) or Tradier brokerage account. Use manual premium input for v1.

Core concepts learned
Exponential backoff retry pattern
with_retry(fn, max_attempts=3) catches anthropic.OverloadedError and APIStatusError(529). Sleeps 2^attempt seconds between retries: 1s then 2s before the 3rd attempt. Shared utility used across all endpoints — no duplication. Mid-stream failures still raise since you can't restart a partial stream.
Two Anthropic clients — timeout separation
client = Anthropic(timeout=60.0) for non-streaming (/chat, /analyze). stream_client = Anthropic(timeout=120.0) for streaming (/analyze-chart, /premarket). Both go through LangSmith wrapping if enabled. Prevents hung calls from blocking Railway workers forever.
Railway keep-alive background task
_keep_alive() coroutine loops every 600s. Checks datetime.now(ET).weekday() < 5 and 9 <=hour < 16 before pinging. Hits http://localhost:{PORT}/ping via httpx.AsyncClient. Spawned at startup via @app.on_event("startup"). Prevents Railway idle cold-starts during market hours.
Claude Code workflow — findings table first
Give Claude Code a detailed prompt listing every file + every issue. It reads the whole repo, produces a findings table (issue #, location, current state, proposed fix), waits for approval, then fixes everything. 8 issues found and fixed across main.py in one session. Show findings → approve → execute is the right pattern.
Claude Chat vs Claude Code — when to use each
Claude Chat = theory, architecture, planning, debugging a snippet. Claude Code = building features across files, refactoring, running commands, fixing multiple bugs at once. Workflow: learn concept in Chat → build it in Claude Code → debrief in Chat. Claude Code has full repo context, can run bash, edit files, verify output.
Stripe skipped — right call for this stage
Stripe makes sense when you have paying users who aren't you. OpenTheDesk is a personal trading tool — adding billing to a solo tool is wasted complexity. Deferred to when OpenTheDesk goes multi-user (Month 5/6 decision). Production hardening shipped instead — more valuable engineering work at this stage.

Core concepts learned
Server-Sent Events (SSE) — instant push vs polling
SSE keeps an open HTTP connection. Server pushes `data: {json}\n\n` instantly when an event fires. Frontend uses EventSource API. 1s heartbeat timeout replaces 30s polling — alerts appear in ~1s instead of up to 30s. Per-client asyncio.Queue in ALERT_SUBSCRIBERS set. Snapshot copy `set(ALERT_SUBSCRIBERS)` on broadcast prevents RuntimeError on concurrent disconnect.
TradingView webhook auth — header vs body secret
TradingView doesn't support custom headers in webhook UI. Workaround: accept secret from X-TV-Secret header OR from JSON body field "secret". Auth passes if either matches TV_WEBHOOK_SECRET env var. Generate secret with `python3 -c "import secrets; print(secrets.token_hex(32))"`.
Pine Script — exact Saty ATR formula
Must use `session=session.extended` in ticker.new() to match Saty's ATR. Regular session only gives ~1pt delta. `ticker_ext = ticker.new(syminfo.prefix, syminfo.ticker, session=session.extended)`. Then `request.security(ticker_ext, "D", close[1], lookahead=barmerge.lookahead_on)`. This is why our levels were clustering wrong before.
Pine Script — A+/A grade system
Every setup has a core condition + grade filter. A+ = all conditions met (conviction + volume + PO zone). A = core only. `plotshape` size: A+ = size.normal, A = size.small. Uppercase text = A+ (GG▲), lowercase = A (gg▲). Webhook payload includes setup, grade, direction fields for backend filtering.
Pine Script — 6 setup conditions
GG: ribbon + trigger crossover + PO neutral. DIV: PO crosses ±100 + price at full ATR + PO turning. FLAG: ribbon 5+ bars + first 13 EMA touch + quiet volume. VOMY: ribbon cross + 13/48 conviction. TWEEZER: matching wicks 2 bars + 50% reclaim. TARGETS: within 5% of GG Open/Complete. STOPS: price reverses through trigger.
React SSE dedup pattern
SSE auto-reconnects on drop. On reconnect, initial GET /alerts loads existing alerts. New SSE message might duplicate one already loaded. Fix: `prev.some(a => a.id === alert.id)` dedup guard before prepending. Prevents double entries without needing server-side dedup.
Alert mark-read — auto vs manual
Bug: bell onClick was calling markAllRead() automatically — so by the time you saw the drawer, everything was read. Fix: removed auto-markAllRead from drawer open. Badge now stays red until you click individual row (markRead) or "Mark all read" button. markRead uses indexOf in newest-first array — clickIdx < seenIdx means alert is newer than baseline.

Per-alert mark-read with Set
Old system used lastSeenId baseline — clicking one alert marked all above it as read. Replaced with readIds: Set<string> stored as JSON array in localStorage (key: tv_alerts_read, capped at 100). markRead(id) adds one id. markAllRead() adds all current ids. alertCount = alerts.filter(a => !readIds.has(a.id)).length.
Unread vs read card states
Unread = full card: accent bar + setup badge + grade badge + price row + ATR level + internals row. Read = compact single line: ticker + timeframe + condition + time only. Dramatically reduces visual noise in the drawer after you've seen an alert.
Alert card color system
Accent bar (4px, alignSelf:stretch) = green for BULL, red for BEAR, gray for TARGET/STOP/DIV. Setup badge = dark green bg (#0f2d1a) + bright green text (#4ade80) for BULL, dark red (#3d0f0f) + red text (#f87171) for BEAR. Grade badge = blue A+ vs gray A. TRIN color: >1.2 red, <0.8 green. ADD color:>+200 green, <-200 red.
Pine Script shape system finalized
Universal rule: ghost (color.new(X, 70)) = A grade, solid = A+ grade. Vomy changed from circle to arrowup/arrowdown — only remaining unused Pine shape. Each setup has unique silhouette: triangle=GG, arrow=Vomy, diamond=Flag, label=DIV, square=Tweezer. Size.normal for all A grades (not size.small) — transparency alone encodes grade.

Journal system — full design
Why Supabase over Google Docs
Google Docs = good for freeform notes, terrible for structured data. Supabase = queryable, filterable, user-specific via Clerk userId, can calculate stats (win rate by setup, avg P&L by time of day, best setup). Already in the stack. Free tier handles this easily.
Database schema — trade_journal table
id (uuid PK), user_id (text, Clerk userId), date (date), ticker (text, default SPX), setup (text: GG/FLAG/VOMY/DIV/TWEEZER), direction (text: BULL/BEAR), entry_price (numeric), exit_price (numeric), contracts (int default 1), pnl (numeric), grade (text: A/A+/B), process_grade (text: how well rules followed), notes (text), internals (jsonb: {trin,add,vold} at entry time), created_at (timestamptz).
Two entry modes
1. Chat command — "I took GG Bear at 7390, exited 7378, 1 contract, made $240" → Claude extracts structured fields via Haiku → POST /journal/entry → Supabase. Fast, no context switching during trading. 2. Manual form in /journal tab — full form with all fields, for end-of-day detailed entries with longer notes.
Claude extraction prompt pattern
User says anything about a trade in chat → detect JOURNAL intent → Haiku extracts: {ticker, setup, direction, entry_price, exit_price, contracts, pnl, grade, notes} as JSON → validate → store. If fields missing, Claude asks only for what's needed. Example: "took GG bear 7390 exit 7378" → entry=7390, exit=7378, pnl auto-calculated as (7390-7378)×contracts×100.
/journal tab UI — three sections
1. Stats row — total trades, win rate %, avg P&L, best setup, current streak. 2. Trade table — date, setup badge, direction, entry, exit, P&L (green/red), grade badge. Sortable, filterable by setup/direction/date/win-loss. 3. Expandable row — click any trade to see full notes + internals snapshot at entry + Claude process grade breakdown.
P&L auto-calculation
SPX options: pnl = (exit_price - entry_price) × contracts × 100. Bear trades: pnl = (entry_price - exit_price) × contracts × 100. If user gives dollar pnl directly, use that. Store both premium P&L and point move for reference. Show running account total on stats row.
AI insights on journal — weekly review
WEEKLY REVIEW command fetches last 5 trading days from Supabase → sends to Sonnet → returns: best setup this week, worst setup, time-of-day performance (morning vs afternoon), process grade trend, specific pattern to fix. Much more powerful than reviewing a Google Doc manually.
Build order — 4 steps
Step 1: Supabase table + RLS policies (user_id = Clerk userId). Step 2: Backend — POST /journal/entry + GET /journal/entries + GET /journal/stats endpoints. Step 3: Claude extraction in /chat — detect trade journal intent, extract fields, store. Step 4: /journal Next.js page — stats row + table + expandable rows.
Key decisions made
Supabase over Google Docs
Structured queryable data beats append-only doc. Stats, filters, AI analysis all require a real DB. Google Doc stays as a free-form scratchpad for rules/context, not trade log.
user_id from Clerk, not hardcoded
Journal is user-specific from day one. Even though it's currently solo use, building with user_id means multi-user works automatically later. Clerk JWT → extract userId → filter all queries by it.
Chat-first entry, form as backup
During live trading you don't want to fill a form. "Exit 7378 profit $240" in chat should just work. Form is for end-of-day detailed retrospective entries with longer notes.
Internals snapshot on entry
When a journal entry is created from chat, fetch live TRIN/ADD/VOLD at that moment and store in internals jsonb field. Builds a historical record of what market character looked like on each trade. Powerful for pattern analysis later.
